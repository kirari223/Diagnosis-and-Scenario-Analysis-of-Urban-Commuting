## 核心答案

### .py vs .ipynb 的选择

**两种方案都可行**，选择取决于你的使用场景：

#### 方案1: .py 文件（推荐用于稳定的、频繁调用的模块）

**优点**:
- ✅ 可以在任何 notebook 中直接 `import` 调用
- ✅ 代码版本管理更清晰（git diff 友好）
- ✅ 函数签名明确，IDE 有代码补全和类型检查
- ✅ 适合多处调用（main_pipeline 第3、4章都要用）

**缺点**:
- ❌ 调试时需要重启 kernel 才能重新加载修改
- ❌ 不能直接看到中间结果的可视化

**适用场景**: 
- 评估模块已经测试稳定
- 需要在 main_pipeline 多个章节中调用
- 代码逻辑复杂，需要类型注解和文档

**实现示例**:
```python
# src/evaluation/pipeline.py

import pandas as pd

    od_matrix: pd.DataFrame,

    output_section: str = None
) -> Dict[str, Any]:
    """评估子流程"""
    # ... 实现代码
    return results

# 在 main_pipeline.ipynb 中调用
from src.evaluation.pipeline import run_evaluation_pipeline
results = run_evaluation_pipeline(od_matrix, zone_gdf, "current_situation")
```

---

#### 方案2: .ipynb 文件（推荐用于开发阶段、需要频繁调整的模块）

**优点**:
- ✅ 可以边写边看中间结果（图表、数据表）
- ✅ 调试方便，修改后直接重新运行 cell
- ✅ 适合出图代码（需要反复调整参数）
- ✅ 可以保留开发过程的注释和说明

**缺点**:
- ❌ 不能直接 `import`，需要用 `%run` 或 `nbconvert` 转换
- ❌ 版本管理不友好（JSON 格式，diff 难读）
- ❌ 跨 notebook 调用需要额外处理

**适用场景**:
- 评估模块还在开发调试阶段
- 出图代码需要频繁调整参数
- 需要保留中间结果的可视化用于检查

**调用方式**:

**方式A: 使用 `%run` 魔法命令**
```python
# main_pipeline.ipynb 中
%run notebooks/sub_pipeline/evaluation_pipeline.ipynb

# 此时 evaluation_pipeline.ipynb 中定义的所有变量和函数都可用
results = run_evaluation_pipeline(od_matrix, zone_gdf)
```

**方式B: 使用 `nbconvert` 转换为 .py 后导入**
```python
# 在 main_pipeline.ipynb 开头
import subprocess
subprocess.run([
    "jupyter", "nbconvert", 
    "--to", "script", 
    "notebooks/sub_pipeline/evaluation_pipeline.ipynb",
    "--output-dir", "src/evaluation"
])

from src.evaluation.evaluation_pipeline import run_evaluation_pipeline
```

**方式C: 使用 `import-ipynb` 库（不推荐，增加依赖）**
```python
import import_ipynb
from notebooks.sub_pipeline import evaluation_pipeline
```

### 各文件夹的职责说明

#### `notebooks/sub_pipeline/` - 分支流程（.ipynb 格式）

**存放内容**:
- 评估模块开发版（`evaluation_dev.ipynb`）
- 可视化模块开发版（`visualization_dev.ipynb`）
- 特定情景的完整流程（如 `scenario1_rigid_full.ipynb`）

**特点**:
- 可以边写边看结果
- 适合频繁调整的代码
- 可以保留开发过程的注释和中间输出
- 使用 `%run` 在 main_pipeline 中调用

**命名规范**:
- 开发版本：`{模块名}_dev.ipynb`（如 `evaluation_dev.ipynb`）
- 完整流程：`{章节名}_{情景名}_full.ipynb`（如 `scenario1_rigid_full.ipynb`）

---

#### `src/evaluation/` - 评估模块（.py 格式）

**存放内容**:
- 稳定的评估指标计算函数（`metrics.py`）
- 评估主流程封装（`pipeline.py`）
- 评估相关可视化（`visualization.py`）

**特点**:
- 代码已测试稳定，不频繁修改
- 可以在多个 notebook 中 `import` 调用
- 有完整的类型注解和文档字符串
- 版本管理友好

**迁移时机**:
- 评估模块在 `evaluation_dev.ipynb` 中测试通过
- 出图参数已确定，不需要频繁调整
- 准备在 main_pipeline 多个章节中复用

---

#### `tempt/` - 测试文件夹（.ipynb 格式）

**存放内容**:
- `01_function_test/`: 单个函数的测试（如测试可达性计算函数）
- `02_module_test/`: 模块级测试（如测试整个评估 pipeline）
- `03_integration_test/`: 集成测试（如测试第3章+评估模块）

**特点**:
- 用于验证代码正确性，不产生论文结果
- 结果输出到 `tempt/tempt_results/`
- 可以随时删除重建，不影响主流程

## 总结：你的情况下的最佳方案

### 推荐方案：先 .ipynb，稳定后可选迁移到 .py

**理由**:
1. ✅ 你的出图代码经常有问题 → .ipynb 方便调试
2. ✅ 时间紧迫 → .ipynb 开发速度快
3. ✅ 评估模块需要在多处调用 → 用 `%run` 即可，不必立即写 .py
4. ✅ 论文截稿前不需要完美架构 → 先跑通，后优化

**存放位置**:
- 开发版本：`notebooks/sub_pipeline/evaluation_dev.ipynb`
- 稳定版本（可选）：`src/evaluation/pipeline.py`

**调用方式**:
- 在 main_pipeline.ipynb 中用 `%run notebooks/sub_pipeline/evaluation_dev.ipynb`
- 修改后重新运行 `%run` 那个 cell 即可生效

**关键提醒**:
- 不要纠结 .py vs .ipynb，够用就行
- 先用 .ipynb 快速开发，稳定后再考虑迁移
- 论文结果正确性 > 代码组织形式
